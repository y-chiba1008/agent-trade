# デモトレードアプリ 実装計画書

## 方針

最小限の機能から動かせる状態を作り、段階的に機能を拡張する。各フェーズは「動くものができる」単位で区切り、フェーズの終わりには必ず手元で動作確認できる状態にする。

フェーズ内の実装順序は目安であり、状況に応じて入れ替えてよい。重要なのは各フェーズの「ゴール」を満たすことである。

---

## Phase 0: 環境構築・土台づくり

**ゴール**: バックエンド（FastAPI）が起動し、PostgreSQLに接続でき、最低限のヘルスチェックAPIが動く状態。

- [ ] Python仮想環境・依存関係管理のセットアップ（poetry / uv 等、お好みのツール）
- [ ] FastAPIの最小構成（`GET /health` 等）
- [ ] PostgreSQLのセットアップ（ローカルDocker等）
- [ ] DBマイグレーションツールの選定・導入（Alembic推奨）
- [ ] プロジェクトのディレクトリ構成決定（サービス層・WebAPI層・MCP層を分離する構成を意識する）

```
backend/
  app/
    api/          # FastAPIのルーター（WebAPI層）
    mcp/          # MCPサーバ実装（MCP層）
    services/     # ビジネスロジック（サービス層）※API/MCP両方から呼ばれる
    models/       # DBモデル
    adapters/     # 価格データソースのアダプタ（取引所ごとの実装を抽象化）
    db/           # DB接続・マイグレーション
```

**検証方法**: `uvicorn`でサーバ起動 → `/health`にアクセスして200が返る。DBにテーブルが1つ作れる。

## Phase 1: 価格取得 + 成行注文のみ（最小限のコア機能）

**ゴール**: BTC-JPYのリアルタイム価格が取得でき、1つのデモ口座に対して成行注文を出すと、DBに口座残高・注文・約定履歴が記録される。フロントエンド・MCPはまだ無し（API単体での動作確認まで）。

### 1-1. 価格取得
- [ ] bitFlyer Realtime API（WebSocket, `lightning_ticker_BTC_JPY`）への接続実装
- [ ] 受信した最新価格をメモリ上に保持する仕組み（シングルトン的な価格ストアでよい）
- [ ] WebSocket切断時の自動再接続処理
- [ ] 価格データソースをアダプタとして抽象化（`PriceSourceAdapter`のようなインターフェースを切り、bitFlyer実装をその1つとする）
- [ ] バックエンドの起動・終了時刻をDBに記録する仕組み（`system_status`テーブル等）を用意する。Phase 3の停止期間リカバリで「前回いつまで稼働していたか」を判定するために必要。

### 1-2. デモ口座管理
- [ ] 口座テーブルの設計・マイグレーション（`accounts`: id, name, balance, created_at等）
- [ ] 口座作成・削除・一覧取得のサービス関数
- [ ] WebAPI: `POST /accounts`, `DELETE /accounts/{id}`, `GET /accounts`

### 1-3. 注文・約定（成行のみ）
- [ ] 注文テーブル・約定テーブルの設計・マイグレーション（`orders`, `executions`）
- [ ] 成行注文受付 → 即時約定 → 残高・ポジション更新のサービス関数
- [ ] WebAPI: `POST /accounts/{id}/orders`（成行注文）, `GET /accounts/{id}/orders`, `GET /accounts/{id}/executions`
- [ ] WebAPI: `GET /price/BTC_JPY`（現在価格取得）

**検証方法**: API単体テスト or curl/Postmanで、口座作成 → 成行注文 → 残高が変動 → 約定履歴が記録される、という一連の流れを確認できる。

## Phase 2: フロントエンド（最小UI） + MCP（最小ツール）

**ゴール**: ブラウザから価格確認・成行注文ができ、AIからもMCP経由で同じ操作ができる。

### 2-1. フロントエンド最小実装
- [ ] Vite + React のプロジェクトセットアップ
- [ ] shadcn/ui の導入（base UIコンポーネント）
- [ ] TanStack Query の導入・APIクライアントのセットアップ
- [ ] Zustand によるクライアント側状態管理のセットアップ（必要なローカル状態が出てきた時点で導入でも可）
- [ ] React Router によるページ構成（口座一覧・口座詳細など）
- [ ] 現在価格の表示（ポーリングまたはWebSocket購読）
- [ ] 口座一覧・作成・削除のUI
- [ ] 成行注文を出すUI（React Hook Form + Zod でバリデーション。買い/売り、数量入力）
- [ ] 注文・約定履歴の簡易表示

### 2-2. MCPサーバ最小実装
- [ ] MCPサーバの起動構成（バックエンドプロセス内で直接ホスト）
- [ ] MCPツール: `get_price`（現在価格取得）
- [ ] MCPツール: `get_account`（口座情報取得）
- [ ] MCPツール: `place_market_order`（成行注文）
- [ ] MCP層はサービス層を呼ぶだけの薄いラッパーとして実装（Phase 1で作ったサービス関数を再利用）

**検証方法**: ブラウザで価格を見ながら成行注文を出せる。Claude Desktop等からMCP経由で同じ口座を操作し、フロントエンドの表示が反映されることを確認する。

## Phase 3: 指値注文・注文監視の仕組み + 停止期間リカバリ

**ゴール**: 指値注文が出せるようになり、価格が条件を満たした時点で自動的に約定する。また、バックエンドが個人PC上で断続的に停止する前提のもと、起動時に停止期間中の価格変動を反映して未約定注文を一括判定できる。

### 3-1. 指値注文・監視ロジック
- [ ] 注文モデルに「注文種別（成行/指値）」「指定価格」「状態（未約定/約定済/キャンセル済）」を追加
- [ ] 価格更新時に未約定の指値注文をチェックする監視ロジックの実装方針を決定する
  - 案A: 価格更新イベントごとに、全口座の未約定注文をDB走査してチェック
  - 案B: 価格更新をイベントとして発行し、監視サービスがサブスクライブする
  - → 個人検証用途であればまずは案A（シンプルな実装）で十分。口座数・注文数が増えた場合に案Bへ移行する判断とする
- [ ] 指値注文の約定処理（約定条件判定 → 約定 → 残高更新）
- [ ] WebAPI / MCPツールに指値注文発行を追加
- [ ] フロントエンドに指値注文UIを追加（指定価格の入力）
- [ ] 未約定注文のキャンセル機能（API/MCP/UI）

### 3-2. 起動時の停止期間リカバリ
- [ ] 起動時、Phase 1で記録した「前回稼働終了時刻」と現在時刻から停止期間を算出する
- [ ] bitFlyer HTTP公開API（`GET /v1/getexecutions`）を使い、停止期間に該当する約定履歴を取得する
  - 件数ベースのページネーション（`before`/`after`/`count`）のため、停止期間の開始時刻をまたぐまで遡って取得し続ける処理を実装する
  - bitFlyerのAPIレート制限に抵触しないよう、リクエスト間に軽いインターバルを設ける
  - 取得対象は生の約定データのままでよく、OHLCなどへの変換は行わない（実装コストを抑えるため）
- [ ] 取得した約定履歴を時系列の昇順に並べ、各時点で未約定の指値注文の条件を満たすか順次判定する一括リカバリ処理を実装する
- [ ] 停止期間が31日を超える場合は、bitFlyer側の制限によりリカバリ不可能な期間が生じる旨をログ等で明示する（エラーにはしない。検証用途として許容する）
- [ ] リカバリ処理の実行タイミングはバックエンド起動シーケンスの先頭（WebSocket接続開始前、または直後）に置き、リカバリ完了後にリアルタイム監視へ切り替える

**検証方法**:
- リアルタイム指値: 現在価格より離れた価格で指値注文を出し、価格がその水準まで動いたときに自動約定することを確認する（テスト時はモック価格データソースを使うと検証しやすい）。
- 停止リカバリ: バックエンドを停止し、数時間〜数日後に再起動した際、その間に価格が指値水準を通過していれば起動直後に約定済みとなっていることを確認する。

## Phase 4: OCO・IFD注文

**ゴール**: より高度な注文（OCO, IFD）に対応し、本番取引所に近い操作感を実現する。

- [ ] 注文同士の関連性（親子関係・グルーピング）をモデル化
- [ ] OCO: 2注文を1組として管理し、片方約定時に他方を自動キャンセルする処理
- [ ] IFD: 親注文約定をトリガーに子注文を自動発行する処理
- [ ] WebAPI / MCPツール / フロントエンドUIへの反映

**検証方法**: OCO注文で一方が約定したらもう一方が自動キャンセルされる。IFD注文で親注文約定後に子注文が自動的に発行される、という一連の挙動をテストする。

---

## Phase 5: ダッシュボード・成績管理の充実

**ゴール**: 口座ごとの成績を可視化し、人間の検証・AIの取引有効性検証の両方に資する情報を提供する。

- [ ] 損益（実現損益・含み損益）の集計
- [ ] 取引履歴の一覧・フィルタリング
- [ ] 口座ごとの成績比較（例: 人間口座とAI口座の損益カーブを並べて表示）
- [ ] 必要に応じて簡易的なチャート表示（価格推移・損益推移）。Recharts（シンプルな折れ線・棒グラフで十分な場合）か lightweight-charts（ローソク足等、本格的なトレード画面に近い見た目が欲しい場合）のどちらを使うかをこの時点で判断する

**検証方法**: 複数口座（人間用・AI用）で取引を行い、ダッシュボード上でそれぞれの成績が正しく集計・表示されることを確認する。

## フェーズ全体の依存関係

```
Phase 0 (環境構築)
   ↓
Phase 1 (価格取得 + 成行注文のコアロジック)
   ↓
Phase 2 (フロントエンド + MCP の最小実装) ← ここで「人間も使える」「AIも使える」状態に到達
   ↓
Phase 3 (指値注文・監視ロジック)
   ↓
Phase 4 (OCO・IFD)
   ↓
Phase 5 (ダッシュボード充実)
```

Phase 2の完了時点で、要件にある「人間用UI」「AI用MCP」の両方が最小限ながら動作する状態になる。これが最初の重要なマイルストーンとなる。
